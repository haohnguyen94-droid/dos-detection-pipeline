# Security Invariants

## Network Isolation
All attack traffic is generated and received within a private Docker bridge
network (`dos-lab`, subnet `172.20.0.0/16`) with no external routing. The
attacker container cannot reach the internet or any host system. Docker's
network namespacing ensures complete isolation between the lab environment
and production networks.

## Least-Privilege Execution
Only the attacker container is granted the `NET_ADMIN` capability (via
`cap_add` in `docker-compose.yml`), which is the minimum privilege required
for Scapy to open raw sockets. The victim and detector containers run with
default (unprivileged) Docker permissions. No container runs as host root
or has access to the host filesystem beyond explicitly mounted volumes.

## Data Privacy
No real user traffic is captured at any stage. Every pcap file is
machine-generated by our own attack scripts against our own victim
container. There is no PII, no external hostname resolution, and no
third-party data involved in any capture or report.

## Synthetic Traffic Only
All three attack types (SYN flood, UDP flood, HTTP Slowloris) and the
benign traffic generator produce fully synthetic packets. Source and
destination IPs are Docker-assigned container addresses. No spoofed
public IPs are used. No DNS queries leave the isolated network.

## Volume Access Control
The shared `pcap-data` volume is mounted read-write in the victim
container (which writes pcaps) and read-only in the detector container
(which reads pcaps). The attacker container has no access to the pcap
volume. This enforces a one-way data flow: victim writes, detector reads,
attacker cannot tamper with captures.

## Tool Responsibility
Attack generation tools (Scapy, raw sockets) are standard security
research utilities used in controlled lab environments. All scripts
include banners identifying the attack type and target, providing an
audit trail in container logs. These tools must not be used outside
of isolated lab environments.
